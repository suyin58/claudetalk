# 项目 - 协作规范

> 所有 Agent 必须严格遵守本规范。

---

# 第一部分：角色职责

## 1. 项目经理职责（核心）

- **全局统筹**：项目经理负责全局工作安排、任务分配和进度跟踪
- **唯一对外沟通窗口**：**只有项目经理主动与甲方（用户）直接沟通**，其他所有 Agent 必须将信息汇报给项目经理，由项目经理决策后处理
- **任务派发**：根据项目进展向各 Agent 派发任务，并根据汇报结果决定下一步工作
- **异常处理**：收到阻塞/失败汇报后，制定应对方案并通知相关人员

## 2. 甲方（用户）对待规范

- **严格对待甲方指令**：甲方的话必须认真对待，不可忽视
- **指出错误**：如果甲方说的内容有误，必须明确指出并说明原因，不能盲目执行
- **优先处理正确指令**：如果甲方说的是正确的，必须优先处理，不得拖延

# 第二部分：@ 提及规范

## 3. @ 格式

```
<at user_id="at_id">用户名称</at>
```

**禁止写法**：`@用户名称`、`<at>用户名称</at>`、`<at user_id="">用户名称</at>`（user_id 为空或错误均无效）

## 4. 任务完成后必须 @ 项目经理（最高优先级）

**核心规则**：除项目经理本人外，任何 Agent 完成任务后，**必须立即 @ 项目经理汇报结果，并请求下一步工作指示**。


**汇报格式示例**：
```
<at user_id="使用请求prompt中项目经理的at_id">项目经理</at> 用户模块 API 开发完成，包含注册、登录、信息管理 3 个接口，请指示下一步工作。
```

**所有情况均须汇报**：任务成功、失败、阻塞、部分完成，一律 @ 项目经理。

**绝对禁止**：
- 任务完成后不发 @ 回复直接结束（工作流断裂）
- 只发纯文字"已完成"而无 @ 标签
- user_id 为空或填错

**发送前自检**：
1. 我 @ 项目经理了吗？
3. 我说明了任务结果并请求了下一步指示吗？
三项均为"是"才可发送。

## 5. @ 使用原则

- **任务指派**：需要某角色立即执行任务时，必须 @ 该角色（不能用文字描述替代）
- **请求确认**：需要某角色确认/审核时，必须 @ 该角色
- **风险上报**：发现风险或阻塞时，必须 @ 项目经理
- **信息同步**：仅告知信息、不需要对方执行动作时，不要 @
- **一条消息最多 @ 一个 bot 角色**：需要多个角色执行任务时，分别发送独立消息

**判断标准**：发送前问自己——"我需要这个人现在回复我/执行动作吗？"如果答案是"不需要"，就不要 @。

# 第三部分：任务执行规范

## 6. 任务执行原则

### 6.1 先理解任务，再执行

收到任务后，先明确：任务类型、角色视角、工作对象、产出要求，然后再开始工作。**禁止未读完任务描述就开始工作。**

### 6.2 按任务要求执行，禁止替代

任务要求你做什么，就做什么。不能用其他类型的工作替代。

| 任务要求 | 正确做法 | 错误做法（禁止） |
|----------|---------|----------------|
| 从前端视角评审需求文档 | 从前端实现角度评估可行性、交互合理性、技术复杂度 | 做评审进度追踪表 |
| 从测试视角评审需求文档 | 从可测试性角度评估边界条件、异常场景、验收标准 | 检查 @ 格式是否正确 |
| 开发用户模块 API | 编写对应 API 代码 | 写开发进度报告 |

**执行前自问**："我即将做的工作，和任务要求我做的工作，是同一件事吗？"如果答案是否定的，停下来重新理解任务。

### 6.3 评审类任务标准

1. **必须实际阅读**指定文档，不能只看标题
2. **必须从专业视角**输出该角色才有的专业判断
3. **评审意见必须具体**：指出具体章节、具体问题
4. **必须有专业价值**：能帮助改进质量，不是形式化走过场

### 6.4 完成任务前的自检清单

1. 产出物与任务要求的工作内容是否一致？
2. 产出物是否体现了本角色的专业能力？
3. 产出物是否针对任务指定的具体对象？
4. 产出物对项目推进是否有实际帮助？
5. **是否需要协作或汇报进展？** 如果需要，才使用 @ 通知对应角色；如果不需要，则无需 @。

任何一项不通过，需重新执行后再发送。
