# 角色分工 × 群聊协商，打造透明可干预的多agent协作框架



多 Agent 系统有很多种搭法，有的靠 Agent 之间互相聊天协商，有的靠中心调度器派活，有的把整个流程画成流程图来跑，有的像蚁群一样自组织……

ClaudeTalk（[项目地址 https://github.com/suyin58/claudetalk](https://github.com/suyin58/claudetalk)）走的是其中一条比较务实的路：**角色分工 + 群聊对话** 的混合模式。

具体来说，它同时借鉴了两种架构思路：

**① 角色分工式（Role-Based）**：系统里有明确的虚拟团队，每个角色有固定的职责边界和系统提示，不会越界干别人的活。任务从用户发起，由主控角色拆解后，按职责链路逐步流转，有点像一个真实的团队在协作。角色是完全开放的，用户可以按自己的行业和业务需求自由定义——ClaudeTalk 内置的是一套软件开发场景的示例角色，但换成法律团队、内容团队、电商运营团队的角色组合同样可以跑起来。

**② 对话协商式（Dialogue-Based）**：这些角色不是在后台默默跑流程，而是真实地"出现"在 IM 群聊里。每个 Agent 以 Bot 身份发言，消息就是协作媒介。Agent 之间通过 @ 对方来传递任务、反馈结果、触发下一步，整个协作过程对人类完全透明，随时可以插手干预。

两者结合的结果是：**流程有序、过程可见、人可随时介入**。既不像纯对话式那样容易跑偏，也不像纯流程图式那样死板，更不像黑盒子一样让人看不懂在干什么。

底层的技术支撑有三层：1）以 IM 群聊作为消息总线，Agent 之间通过 @ 消息传递任务，用户可以随时干预；2）以 Claude Code SubAgent 作为每个角色的执行引擎，支持为不同 Agent 独立配置 Prompt 和模型；3）三层记忆机制保证记忆不丢失、减少 Token 消耗、避免上下文腐烂：
- **群聊历史消息是协作记忆**：群聊记录同时作为上下文记忆——每个 Agent 收到指令时，会获取群聊的部分历史对话，用于理解当前协作的来龙去脉
- **文件系统是长期记忆**：所有 Agent 共享项目文件（需求文档、架构设计文档、测试报告等），形成隐式的数据契约，跨 Agent、跨轮次的信息都沉淀在这里
- **Session 是短期记忆**：Claude Code 的 `--resume` 机制保证每个 Agent 自身的对话连续性，不会"失忆"

整体上，它不是一个黑盒的自动化流水线，而是一个**透明的、可观测的、人类可随时干预的协作系统**。

ClaudeTalk 的 Multi-Agent 系统本质上是：**以 IM 群聊为总线，以 Claude Code SubAgent 为执行单元，以共享文件系统为记忆载体**，构建的一套可强干预的多 Agent 自主协作框架。

## 二、Agent 角色体系

角色体系是完全开放的，用户可以通过 `agent_auto_config.json` 自定义任意角色，每个角色对应一个独立的 Claude Code SubAgent 进程。ClaudeTalk 内置了一套**软件开发场景的示例角色**，方便开箱即用：

| 角色 | 英文名 | 核心职责 | 触发方式 |
|------|--------|---------|---------|
| 项目经理 | Project Manager | 任务分解、里程碑管理、最终决策 | 在群里被任意角色或者成员 @ |
| 需求分析师 | Requirements Analyst | 需求文档、验收标准 | 在群里被任意角色或者成员 @ |
| 系统架构师 | System Architect | 技术选型、架构设计、API 规范 | 在群里被任意角色或者成员 @ |
| 前端工程师 | Frontend Engineer | UI 实现、原型开发 | 在群里被任意角色或者成员 @ |
| 后端工程师 | Backend Engineer | API 开发、数据库脚本 | 在群里被任意角色或者成员 @ |
| 代码审查员 | Code Reviewer | 代码质量、安全审查 | 在群里被任意角色或者成员 @ |
| 测试工程师 | QA Engineer | 单测/集成/E2E 测试 | 在群里被任意角色或者成员 @ |
| 运维工程师 | DevOps Engineer | 构建、部署、回滚 | 在群里被任意角色或者成员 @ |

> 这套角色只是示例，不是限制。你完全可以把它换成适合自己业务的角色组合，比如：
> - **内容团队**：选题策划 → 文案撰写 → 编辑审核 → 发布运营
> - **法律行业**：案件分析师 → 合规审查员 → 文书起草员
> - **电商团队**：选品分析师 → 定价策略师 → 营销文案 → 数据复盘
>
> 只要能用文字描述清楚一个角色的职责和边界，就能把它配置进来。

## 三、协作机制：@ 驱动的事件总线

### 3.1 核心设计思路

Multi-Agent 的协作**不依赖任何中心调度器**，而是以 **IM 群聊的 @ 消息作为事件总线**：

- 每个 Agent 只监听发给自己的 @ 消息
- 完成任务后，通过 @ 下一个 Agent 触发流转
- 项目经理是唯一的"总控"角色，负责任务分解和最终汇报

```
用户
 │
 ▼ @项目经理
项目经理 ──@需求分析师──▶ 需求分析师（输出 requirements.md）
         │
         └──@系统架构师──▶ 系统架构师（输出 architecture.md, api_spec.md）
                  │
                  ├──@前端工程师──▶ 前端工程师
                  │                    │ @代码审查员
                  └──@后端工程师──▶ 后端工程师
                                       │ @代码审查员
                                       ▼
                                  代码审查员
                                       │ @测试工程师
                                       ▼
                                  测试工程师
                                       │ @运维工程师（部署本地环境）
                                       │ @运维工程师（部署测试环境）
                                       ▼
                                  运维工程师
                                       │ @项目经理（等待甲方确认）
                                       ▼
                                  [用户确认] ──@运维工程师──▶ 线上部署
```

### 3.2 @ 规范的技术实现

每个 Agent 的 Prompt 中都强制注入了 `CLAUDE.md` 协作规范，规定了 @ 的 XML 格式：

```xml
<at user_id="从at_id映射表中获取正确的at_id">用户名称</at>
```

`at_id` 映射表通过 `context-message.template` 中的 `{{chatMembersSection}}` 变量注入，在每次消息处理时动态填充群成员信息，确保 Agent 能找到正确的 `user_id`。

### 3.3 防死循环机制

为避免 Agent 之间无限互相 @ 造成死循环，系统设计了**修复轮次上限**：

```
代码审查员 ←→ 后端工程师（最多 3 轮修复）
测试工程师 ←→ 开发工程师（最多 3 轮修复）

超过 3 轮 → 强制上报 @项目经理 决策
```

同时，`context-message.template` 在每次消息中注入历史上下文，Agent 可以感知历史回复，避免重复触发。

## 四、消息上下文构建：context-message.template

每次 Agent 收到消息时，Channel 层会将原始消息加工成结构化的上下文 Prompt，再发给 Claude Code：

```
## 工作流程
请按照以下步骤处理消息：
1. 优先处理当前消息
2. 参考历史上下文
...

## 群聊上下文(at_id映射表)
{{chatMembersSection}}     ← 群成员列表 + at_id 映射

### 当前消息（优先处理）
- 发送者: {{senderInfo}}
- 消息内容: {{messageText}}
{{mentionsSection}}        ← 被 @ 的人员列表

### 历史消息（按时间倒序，仅供参考）
{{historySection}}         ← 最近 N 条群聊历史
```

这个模板是 Agent 感知"谁在说话、说了什么、历史上下文是什么"的核心机制。

## 五、记忆机制

ClaudeTalk 的记忆分为三个层次：

### 5.1 短期记忆：Session 持久化

每个 Agent 的对话 Session 通过 `claude -p --resume <session_id>` 实现多轮连续对话，Session 信息持久化到：

```
{workDir}/.claudetalk-sessions.json
```

Session Key 的设计保证了多 Agent、多 Channel、多项目完全隔离：

```
sessionKey = conversationId + "\x00" + workDir + "\x00" + profile + "\x00" + channel
```


### 5.2 中期记忆：群聊历史注入

Channel 层在每次构建消息上下文时，会将最近 N 条群聊历史注入到 `{{historySection}}`，让 Agent 能感知近期的协作上下文，即使 Session 重置也不会完全失忆。

### 5.3 长期记忆：共享文件系统

这是 Multi-Agent 协作中最关键的记忆机制——**所有 Agent 共享同一个项目文件系统**：

```
docs/
  requirements.md          ← 需求分析师写入，所有人读取
  architecture.md          ← 架构师写入，开发/测试/运维读取
  api_spec.md              ← 架构师写入，前后端读取
  database_design.md       ← 架构师写入，后端读取
  code_review_report.md    ← 代码审查员写入
  test_report.md           ← 测试工程师写入，运维读取
  deploy_log.md            ← 运维工程师维护
database/
  migrations/              ← 后端工程师写入
  seeds/                   ← 后端工程师写入
tests/                     ← 测试工程师写入
```

每个 Agent 的 Prompt 中明确规定了**读哪些文件、写哪些文件**，形成了一套隐式的"数据契约"，避免 Agent 之间的信息孤岛。

## 六、SubAgent 的技术实现

### 6.1 SubAgent 的启动方式

ClaudeTalk 通过 `--agents` 参数将角色定义注入 Claude Code：

```bash
claude -p \
  --output-format json \
  --dangerously-skip-permissions \
  --agents '{"后端工程师": {"description": "...", "prompt": "...", "model": "claude-sonnet-4-5"}}' \
  --resume <session_id>
```

当 SubAgent 模式启用时，消息会被包装成：
```
Use the 后端工程师 agent to handle this: <实际消息内容>
```

## 七、关键设计决策与权衡

### 7.1 为什么用 IM 群聊而不是 API 调用链？

| 方案 | 优点 | 缺点 |
|------|------|------|
| IM 群聊总线 | 人类可实时观察、随时干预、天然的异步解耦 | 依赖 IM 平台 |
| API 调用链 | 速度快、无外部依赖 | 黑盒运行、难以干预、调试困难 |

ClaudeTalk 选择 IM 群聊，核心原因是**可观测性和可干预性**——人类可以随时在群里发消息打断、纠正、重定向 Agent 的行为。

### 7.2 为什么用文件系统而不是数据库作为共享记忆？

- Claude Code 天然具备文件读写能力，无需额外工具
- 文件内容对人类直接可读，便于审查和干预
- 文件路径即"数据契约"，Agent 之间通过约定路径传递信息，简单可靠

### 7.3 Session 隔离策略

Session Key 中包含 `workDir + profile + channel`，确保：
- 同一个 IM 群里的不同 Agent 互不干扰
- 同一个 Agent 在不同项目目录下的 Session 完全隔离
- 飞书和钉钉的 Session 不会混用

## 八、完整协作流程示例

以"开发用户登录功能"为例：

```
用户（群聊）
  │ @项目经理 请开发用户登录功能
  ▼
项目经理
  │ 分解任务，读取项目现状
  │ @需求分析师 请输出用户登录的需求文档
  ▼
需求分析师
  │ 写入 docs/requirements.md
  │ @项目经理 需求文档已完成
  ▼
项目经理
  │ @系统架构师 请设计登录接口规范
  ▼
系统架构师
  │ 写入 docs/api_spec.md, docs/database_design.md
  │ @项目经理 接口规范已完成
  ▼
项目经理
  │ @后端工程师 请实现用户登录接口
  ▼
后端工程师
  │ 读取 docs/requirements.md, docs/api_spec.md, docs/database_design.md
  │ 写入 database/migrations/001_create_users_table.sql
  │ 写入 src/api/user.controller.js
  │ @代码审查员 请审查用户登录接口
  ▼
代码审查员
  │ 读取 docs/architecture.md, src/api/user.controller.js
  │ 写入 docs/code_review_report.md
  │ @后端工程师 发现 SQL 注入风险，请修复
  ▼
后端工程师（修复）
  │ @代码审查员 已修复，请复审
  ▼
代码审查员（复审通过）
  │ @测试工程师 代码审查通过，请测试
  ▼
测试工程师
  │ 读取 docs/requirements.md, docs/api_spec.md
  │ 写入 tests/user.test.js
  │ 执行测试
  │ @运维工程师 测试通过，请部署本地环境
  ▼
运维工程师（本地部署）
  │ @测试工程师 本地环境已就绪
  ▼
测试工程师（E2E 测试）
  │ 写入 docs/test_report.md
  │ @运维工程师 所有测试通过，请部署测试环境
  ▼
运维工程师（测试环境部署）
  │ @项目经理 测试环境已就绪，请验收
  ▼
项目经理
  │ （通知用户验收）
  ▼
用户确认上线
  │ @项目经理 同意发布
  ▼
项目经理
  │ @运维工程师 甲方已同意，请部署线上
  ▼
运维工程师（线上部署）
  │ @项目经理 线上部署完成 ✅
```

## 九、总结

ClaudeTalk 的 Multi-Agent 架构由三个核心支柱构成：

**① 多角色 Agent 体系**：系统内置了一套完整的虚拟团队——项目经理、需求分析师、系统架构师、前后端工程师、代码审查员、测试工程师、运维工程师，每个角色有明确的职责边界和独立的 Prompt 配置。角色体系完全开放，可以按业务需求自由定制，换成法律团队、内容团队、电商运营团队同样可以跑起来。

**② IM 群聊是神经网络**：Agent 之间不依赖任何中心调度器，而是以 IM 群聊的 @ 消息作为事件总线——每个 Agent 只响应发给自己的 @ 消息，完成任务后通过 @ 下一个 Agent 触发流转。整个协作过程在群聊中实时可见，人类随时可以发消息打断、纠正、重定向 Agent 的行为。

**③ 三层记忆系统**：
- **短期记忆（Session）**：Claude Code 的 `--resume` 机制保证每个 Agent 自身的对话连续性，不会"失忆"
- **中期记忆（群聊历史）**：每次消息处理时注入最近 N 条群聊历史，让 Agent 感知近期协作上下文，即使 Session 重置也不会完全失忆
- **长期记忆（共享文件系统）**：所有 Agent 共享同一个项目文件系统，需求文档、架构设计、测试报告等沉淀在文件里，形成跨 Agent、跨轮次的隐式数据契约

这套架构的最大价值在于：**它不是一个黑盒的自动化流水线，而是一个透明的、可观测的、人类可随时干预的协作系统**。
