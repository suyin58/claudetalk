# ClaudeTalk 多 Agent 能力：给不同角色配独立的 Claude

上篇说了怎么把钉钉接进 Claude Code，这篇聊聊最近加的一个新功能——多 Agent 支持。

## 为什么要搞多 Agent

先说背景。ClaudeTalk 跑起来之后，我自己用得挺顺，但很快遇到一个问题：同一个项目，我有时候想让 Claude 帮我写代码，有时候想让它帮我整理需求文档。这两件事对 Claude 的"人设"要求完全不一样。

写代码的时候，我希望它专注技术，不要废话，直接给方案。整理文档的时候，我希望它更偏产品视角，帮我想用户故事、梳理边界条件。

最开始的做法是在 `systemPrompt` 里写一大段描述，但这有个问题：你没法在同一个钉钉会话里切换角色，因为 session 是绑定的。要切换就得发 `/new` 清掉上下文，很麻烦。

更大的问题是：不同角色的**权限**应该不一样。PM 角色理论上只需要读文件、改文档，不应该能跑 `npm run build`；Dev 角色需要完整的工具权限。用 `systemPrompt` 是没法控制这个的，Claude 该用什么工具还是用什么工具。

所以就有了多 Agent 的需求。

## Claude Code 的 SubAgent 机制

在动手之前，先搞清楚 Claude Code 原生支持什么。

Claude Code 有一个 SubAgent 机制，本质上是把角色定义写成 Markdown 文件，放在 `.claude/agents/` 目录下。文件格式是带 YAML frontmatter 的 Markdown：

```markdown
---
name: "dev"
description: "开发工程师角色"
model: "claude-sonnet-4-6"
permissions:
  allow:
    - "Read(./**)"
    - "Edit(./src/**)"
    - "Bash(npm test)"
  deny:
    - "Bash(npm publish)"
---

你是开发工程师，专注技术实现，确保代码质量。
```

这个机制有几个地方值得注意：

**权限是真实生效的**，不是靠 prompt 约束的。`deny` 里的规则会在工具调用层面拦截，不是让 Claude "尽量不要做"，而是真的做不了。

**模型可以单独指定**。这意味着你可以给 PM 角色配 Haiku（便宜），给 Dev 角色配 Sonnet（能力强），按需分配算力。

**项目级和用户级都支持**。放在项目的 `.claude/agents/` 下，团队成员都能用；放在 `~/.claude/agents/` 下，只有自己用。

调用方式有两种：
- 自动委托：Claude Code 根据上下文自己判断用哪个 SubAgent
- 显式调用：通过 `--agent` 参数指定

## 设计思路

整体方案想清楚了再动手。核心约束是：**不破坏现有配置结构**。已经在用 ClaudeTalk 的人，不应该因为这个新功能而需要改配置。

所以设计上分三层：

```
有 profile + 启用 SubAgent  →  显式调用 SubAgent（--agent 参数）
有 profile + 未启用 SubAgent →  传统 systemPrompt 方式（--append-system-prompt）
无 profile                  →  自动委托（不传额外参数，Claude 自己选）
```

配置文件结构也保持向后兼容，SubAgent 相关字段是可选的：

```json
{
  "profiles": {
    "pm": {
      "DINGTALK_CLIENT_ID": "PM 机器人 AppKey",
      "DINGTALK_CLIENT_SECRET": "PM 机器人 AppSecret",
      "systemPrompt": "你是产品经理，负责需求分析",
      "subagentEnabled": true,
      "subagentModel": "claude-haiku-4-5",
      "subagentPermissions": {
        "allow": ["Read(./**)"],
        "deny": ["Edit(./src/**)"]
      }
    }
  }
}
```

没有 `subagentEnabled` 字段的 profile，行为和之前完全一样。

## 实现细节

### 自动生成 SubAgent 文件

配置完 profile 之后，需要在 `.claude/agents/` 下生成对应的 Markdown 文件。这个过程是自动的，用户不需要手动写。

```typescript
async function createSubagentFile(
  profileName: string,
  workDir: string,
  systemPrompt?: string,
  model?: string,
  permissions?: SubagentPermissions
): Promise<void> {
  const agentsDir = join(workDir, '.claude', 'agents')
  if (!existsSync(agentsDir)) {
    mkdirSync(agentsDir, { recursive: true })
  }

  const agentFile = join(agentsDir, `${profileName}.md`)

  const lines: string[] = ['---']
  lines.push(`name: "${profileName}"`)
  lines.push(`description: "ClaudeTalk 角色: ${profileName}"`)
  if (model) lines.push(`model: "${model}"`)

  if (permissions) {
    lines.push('permissions:')
    if (permissions.allow?.length) {
      lines.push('  allow:')
      permissions.allow.forEach(rule => lines.push(`    - "${rule}"`))
    }
    if (permissions.deny?.length) {
      lines.push('  deny:')
      permissions.deny.forEach(rule => lines.push(`    - "${rule}"`))
    }
  } else {
    // 没有自定义权限时给一个合理的默认值
    lines.push('permissions:')
    lines.push('  allow:')
    lines.push('    - "Read(./**)"')
    lines.push('    - "Edit(./**)"')
    lines.push('    - "Bash(npm test)"')
    lines.push('    - "Bash(npm run build)"')
    lines.push('  deny:')
    lines.push('    - "Bash(rm -rf *)"')
    lines.push('    - "Bash(npm publish)"')
  }

  lines.push('---')
  lines.push('')
  lines.push(systemPrompt ?? `你是 ${profileName} 角色，负责相关工作。`)

  writeFileSync(agentFile, lines.join('\n') + '\n', 'utf-8')
}
```

### Session 隔离和配置变更检测

这里有个细节要处理：如果用户改了配置（比如把 `subagentEnabled` 从 false 改成 true），旧的 session 是用 `systemPrompt` 方式建立的，继续用会有问题。

解决方案是在 session 记录里存一个 `subagentEnabled` 字段，每次收到消息时和当前配置对比：

```typescript
interface SessionEntry {
  sessionId: string
  lastActiveAt: number
  isGroup: boolean
  conversationId: string
  userId: string
  subagentEnabled: boolean  // 记录建立 session 时的配置
}
```

检测逻辑：

```typescript
const currentConfig = loadConfig(workDir, profile)
const currentSubagentEnabled = currentConfig?.subagentEnabled ?? false

if (existingEntry && existingEntry.subagentEnabled !== currentSubagentEnabled) {
  // 配置变了，清掉旧 session，用新配置重建
  sessionMap.delete(sessionKey)
  saveSessionMap()
  return callClaude(/* 递归调用，走新建 session 的逻辑 */)
}
```

这个设计的代价是：配置变更后，之前的对话上下文会丢失。但我觉得这是合理的取舍——如果你把 PM 角色改成了 Dev 角色，之前那段对话的上下文本来就不应该带过来。

Session key 的生成也包含了 profile 信息：

```typescript
function getSessionKey(conversationId: string, workDir: string, profile?: string): string {
  return profile
    ? `${conversationId}|${workDir}|${profile}`
    : `${conversationId}|${workDir}`
}
```

这样 `pm` 和 `dev` 两个角色的 session 天然隔离，互不干扰。

### 交互式配置引导

配置流程加了一段 SubAgent 引导，在现有的 `clientId`、`clientSecret`、`systemPrompt` 配置之后：

```
🤖 SubAgent 配置（推荐）
   SubAgent 是 Claude Code 的原生角色机制，可以提供更精细的权限控制和模型选择。
   如果不配置，将使用传统的 systemPrompt 方式。
是否配置 SubAgent？(Y/n): Y
  模型 (默认: claude-sonnet-4-6): claude-haiku-4-5
  是否自定义权限？(y/N): N

✅ 角色 [pm] 配置已保存到 ./.claudetalk.json
✅ SubAgent 文件已创建: ./.claude/agents/pm.md
```

## 实际使用效果

配置完之后，启动不同角色的机器人：

```bash
# 终端 1：PM 角色，用 Haiku 模型，只读权限
claudetalk --profile pm

# 终端 2：Dev 角色，用 Sonnet 模型，完整权限
claudetalk --profile dev
```

两个钉钉机器人，对应两个完全独立的 Claude 实例，各自有独立的会话记忆和权限边界。

PM 机器人发 `npm run build` 相关的指令，Claude 会告诉你没有权限；Dev 机器人可以正常执行。这个权限控制是在工具调用层面实现的，不是靠 prompt 约束，所以比较可靠。

## 一些没想清楚的地方

**SubAgent 文件的版本管理**：生成的 `.claude/agents/pm.md` 要不要提交到 git？提交的话团队成员可以共享配置，但 `systemPrompt` 里可能有一些个人化的内容。目前没有强制要求，看团队自己决定。

**权限配置的引导**：现在的交互式配置只支持选"是否自定义权限"，如果选是，还没有做完整的引导流程，需要用户手动编辑配置文件。这块后续可以做得更细。

**SubAgent 文件不存在时的降级**：如果 `.claude/agents/pm.md` 被删了，但配置里 `subagentEnabled` 还是 true，会怎样？目前的处理是降级到 `systemPrompt` 方式，但这个降级逻辑还需要更完善的错误处理。

## 总结

多 Agent 这个功能本质上是把 Claude Code 原生的 SubAgent 机制暴露出来，让 ClaudeTalk 的用户能用上。核心价值有两个：

一是**权限隔离**，不同角色真实的工具权限不同，不是靠 prompt 约束；二是**模型选择**，可以按角色的实际需求分配模型，控制成本。

对于大多数个人用户，`systemPrompt` 方式已经够用了。SubAgent 更适合团队场景，或者对权限控制有明确需求的情况。
