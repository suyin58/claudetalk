# 群聊即工作台：用飞书群实现多 Agent 自主协作

前两篇分别讲了怎么把钉钉/飞书接进 Claude Code，以及怎么给不同角色配独立的 Claude。这篇聊一个更有意思的场景——让机器人之间自己对话。

## 从一个真实的需求说起

我在用 ClaudeTalk 的时候，遇到过这样的情况：

产品经理（PM 机器人）整理完需求，需要让前端开发（fdev 机器人）评估工作量。按之前的玩法，我得手动把 PM 的输出复制一遍，再发给 fdev。这个"人肉传话"的过程很烦，而且容易出错。

更理想的状态是：PM 机器人说完，直接 @ 前端开发机器人，前端开发机器人自己接过去处理。我在旁边看着，觉得哪里不对随时插一句。

这就是这篇要讲的东西。

## 飞书群是天然的多 Agent 工作台

仔细想想，飞书群本身就是一个很好的多 Agent 协作场所：

- **消息可见**：所有机器人的发言都在群里，人工可以完整看到整个协作过程
- **随时干预**：觉得某个机器人跑偏了，直接 @ 它说"不对，重新来"
- **角色清晰**：每个机器人有自己的名字和头像，谁说了什么一目了然
- **历史可查**：整个协作过程留在群消息里，随时可以回溯

这比那种"Agent 在后台自己跑，你只能看最终结果"的方式直观多了。

## 一个绕不开的平台限制

飞书有个限制：**机器人发的消息，其他机器人收不到**。这是平台层面的设计，防止机器人之间无限循环触发。

所以没法靠飞书的事件推送来实现机器人间通信。得换个思路。

解决方案是用**共享文件**做消息队列。几个机器人跑在同一台机器上，通过读写本地文件来传递消息。飞书群负责展示和人工干预，文件负责机器人之间的实际通信。

## 具体怎么运作

整个流程分两段：

**第一段：PM 机器人发消息**

PM 机器人处理完用户的需求，生成回复，回复里 @ 了前端开发机器人：

```
PM agent 回复如下：

<at user_id="cli_a9408db378b8dbce">前端开发</at> 这个需求的前端部分需要你评估一下工作量，主要涉及...
```

消息发到飞书群之后，ClaudeTalk 检测到回复里有 @ 标签，找到被 @ 的机器人（根据 `user_id` 匹配 `chat-members.json` 里记录的机器人信息），把这条消息写进 `bot_前端开发.json`：

```json
{
  "id": "c17ae893-22c3-4fe9-9fc0-2ef1ae0ae3d9",
  "from": "pm",
  "chatId": "oc_bd882a5aa77d348e689aa7dc0325ebfa",
  "messageId": "om_x100b538a770efd60b29e4db9b941871",
  "message": "PM agent 回复如下：\n\n<at user_id=\"cli_a9408db378b8dbce\">前端开发</at> ...",
  "createdAt": 1774969756366
}
```

**第二段：前端开发机器人接单**

前端开发机器人每 5 秒扫一次自己的 `bot_前端开发.json`。发现有新消息，且消息写入已超过 10 秒（留出时间让飞书群消息稳定），就开始处理：

1. 先给原消息回复一个 👌 表情，表示"收到了"
2. 把消息内容交给 Claude CLI 处理
3. 处理完把回复发到飞书群
4. 从文件里删掉这条已处理的消息

整个过程在飞书群里看起来就是：PM 机器人说完，前端开发机器人的消息上出现一个 👌，然后前端开发机器人开始回复。

## 底层用的是 Claude Code 的 SubAgent

每个机器人背后是一个独立的 Claude Code 实例，配置了各自的 SubAgent：

```markdown
---
name: "fdev"
description: "前端开发工程师"
model: "claude-sonnet-4-5"
permissions:
  allow:
    - "Read(./**)"
    - "Edit(./src/**)"
  deny:
    - "Bash(npm publish)"
---

你是前端开发工程师，擅长 React 和 TypeScript，负责前端功能实现和工作量评估。
```

SubAgent 的权限是真实生效的，不是靠 prompt 约束。前端开发机器人只能读写 `src/` 目录，不能发布包。PM 机器人只读，不能改代码。这个边界在工具调用层面就拦住了，Claude 想越界也做不到。

## 人工干预的几种方式

这套机制里，人始终是在场的，不是完全甩手：

**直接插话**：机器人在群里讨论的时候，你随时可以 @ 某个机器人说"等等，这个方向不对"，机器人会把你的消息当作新的输入处理。

**接管某个角色**：如果觉得某个机器人的判断有问题，可以直接在群里说出你的判断，让另一个机器人基于你的判断继续往下走。

**终止协作**：发 `/new` 给某个机器人，清掉它的会话记忆，重新开始。

**观察整个过程**：所有机器人的发言都在群消息里，你可以完整看到它们是怎么协作的，哪个环节出了问题一眼就能看出来。

## 使用前提

要让这套机制跑起来，有几个前提：

1. **同一台机器**：多个 ClaudeTalk 实例需要运行在同一台机器上，通过本地文件通信
2. **同一个飞书群**：所有机器人都得在同一个群里，消息路由才能正确
3. **机器人信息已记录**：被 @ 的机器人需要在 `chat-members.json` 里有记录（首次在群里发消息后会自动积累）

启动方式和之前一样，每个角色开一个终端：

```bash
# 终端 1：PM 机器人
claudetalk --profile pm

# 终端 2：前端开发机器人
claudetalk --profile fdev
```

## 这套方案的边界

说几个目前还没解决的问题，免得用完踩坑：

**只支持同机部署**。如果两个机器人跑在不同机器上，文件共享就不行了。跨机器的场景需要换成 Redis 或者 HTTP 接口，目前还没做。

**消息有 10 秒延迟**。这是故意设计的，给飞书群消息留出稳定时间。如果对实时性要求很高，这个延迟会比较明显。

**机器人间的循环调用没有保护**。如果 A 机器人的回复里 @ 了 B，B 的回复里又 @ 了 A，会无限循环。目前靠 prompt 约束，没有在代码层面做防护。

## 总结

这套方案的核心思路是：**用飞书群做展示层，用本地文件做通信层**。飞书群解决了"人工可见、随时干预"的问题，本地文件绕过了飞书平台不让机器人互相收消息的限制。

底层的 Claude Code SubAgent 机制保证了每个角色有独立的上下文、独立的权限边界，不会互相干扰。

整体来说，这更像是一个"有人在场的半自动协作"，而不是完全自主的 Agent 系统。机器人负责执行，人负责把控方向。这个定位在实际工作中反而更实用——你不需要完全信任 Agent，但可以让它帮你省掉大量重复的传话和格式化工作。
